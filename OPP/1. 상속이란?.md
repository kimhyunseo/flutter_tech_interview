# 상속이란?
상속은 객체지향 프로그래밍(OOP)에서
기존 클래스의 속성과 메서드를 물려받아 새로운 클래스를 만드는 개념

> "이미 만들어둔 설계를 재사용해서 더 구체적인 걸 만드는 것"

## 상속을 사용하는 이유
### 코드의 재사용성 (DRY: Don't Repeat Yourself)
똑같은 코드를 여러 번 작성하는 것만큼 비효율적인 일은 없음
이유: 공통된 속성(데이터)과 메서드(기능)를 부모 클래스에 한 번만 정의해두면, 자식 클래스들은 이를 그대로 가져다 쓸 수 있음

### 유지보수의 편의성
수정 사항이 생겼을 때 고쳐야 할 곳을 최소화
이유: 공통 로직에 버그가 있거나 기능을 변경해야 할 때, 부모 클래스 한 곳만 수정하면 이를 상속받는 모든 자식 클래스에 즉시 반영
효과: 수십 개의 파일을 일일이 찾아다니며 수정하다가 발생할 수 있는 실수를 원천 봉쇄

### 다형성(Polymorphism) 구현
하나의 타입으로 여러 형태의 객체를 다룰 수 있게 함
이유: 부모 타입의 변수에 자식 객체를 담을 수 있음
예시: List<ChatRoom>이라는 리스트 하나에 NormalChatRoom과 StadiumChatRoom을 섞어서 담을 수 있음. 화면을 그릴 때는 "어떤 방이든 상관없이 enter() 메서드를 실행해!"라고 명령할 수 있음

### 계층적 구조화를 통한 가독성 향상
프로젝트의 전체적인 구조를 논리적으로 파악하기 쉬워짐 (is-a 관계)
이유: 클래스 간의 관계가 명확해지면, 코드를 처음 보는 사람도 "아, 이 클래스들은 모두 이 부모의 성격을 물려받았구나"라고 금방 이해할 수 있음
효과: 무분별하게 흩어진 코드보다 관리 포인트가 명확해짐

### 기능의 확장 (Extensibility)
기존의 기능을 해치지 않으면서 새로운 기능을 추가하기 좋음
이유: 이미 잘 작동하고 있는 부모 클래스는 건드리지 않고, 새로운 자식 클래스를 만들어 필요한 기능만 덧붙이거나(Override) 추가할 수 있음
효과: 기존 코드의 안정성을 유지하면서 새로운 요구사항을 빠르게 반영할 수 있음

## 상속의 단점
### 강한 결합도 (Tight Coupling)
속은 부모 클래스와 자식 클래스 사이를 아주 끈끈하게 연결
문제: 부모 클래스의 내부 로직을 조금만 수정해도, 이를 상속받은 모든 자식 클래스가 오작동할 위험이 있음. 이를 **'깨지기 쉬운 기반 클래스(Fragile Base Class) 문제'** 라고 함

### 불필요한 기능의 상속 - "고릴라와 바나나 문제"
객체지향의 거장 조지 셀리(Joe Armstrong)는 이를 이렇게 비유했습니다.
> "당신은 바나나 하나를 원했지만, 바나나를 들고 있는 고릴라와 그 뒤의 정글 전체를 통째로 넘겨받게 된다."

문제: 자식 클래스에게는 전혀 필요 없는 부모 클래스의 메서드나 변수들까지 모두 물려받아 메모리를 차지하고 코드를 무겁게 만듦

### 유연성 부족 (Inflexibility)
상속 관계는 **컴파일 타임(코드 작성 시점)** 에 결정되며, 실행 중에 바꿀 수 없음

문제: 한 번 상속을 받으면 그 관계를 끊기가 매우 어려움. 클래스 계층 구조가 깊어질수록 나중에 구조를 변경하는 것은 거의 불가능

## 상속의 한계를 극복하는 방법
### 합성 (Composition)
> "상속받지 말고 가져다 써라"
상속이 "A는 B다(Is-a)"라면, 합성은 **"A는 B를 가지고 있다(Has-a)"**는 개념
필요한 기능을 별도의 클래스로 만들고, 이를 부모-자식 관계가 아닌 '부품'처럼 가져와 사용하는 방식

해결책: 부모의 모든 것을 강제로 물려받지 않고, 필요한 기능만 주입받아 사용

### 인터페이스 (Interface/Abstract Class)
> "행동의 규칙만 정해라"
구체적인 구현 내용을 물려주는 대신, "이런 기능을 반드시 구현해야 해"라는 **설계도(규칙)** 만 넘겨주는 방식
해결책: 부모의 내부 구현이 바뀌어도 자식에게 영향이 적습니다. 자식은 규칙에 맞게 자신만의 코드를 짜면 되기 때문

### 믹스인 (Mixin)
> "필요한 기능만 레고처럼 조립해라"
Dart에서 상속의 단점을 해결하는 가장 강력한 무기
부모-자식의 깊은 계층 구조를 만들지 않고, 특정 기능만 떼어내어 여러 클래스에 붙일 수 있음
해결책: 다중 상속의 효과를 내면서도 구조를 단순하게 유지
