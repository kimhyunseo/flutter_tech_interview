# SOLID 원칙은 무엇인가요?
**시간이 지나도 유지보수가 쉽고, 확장이 유연한 소프트웨어"** 를 만들기 위한 일종의 가이드라인

## SRP: 단일 책임 원칙 (Single Responsibility Principle)
개념: 하나의 클래스는 **단 하나의 책임(기능)**만 가져야 함
이유: 클래스가 너무 많은 일을 하면, 코드 하나를 고칠 때 어디가 터질지 모름

## OCP: 개방-폐쇄 원칙 (Open/Closed Principle)
개념: 확장에는 열려 있어야 하고, 수정에는 닫혀 있어야 한다.
이유: 새로운 기능을 추가할 때 기존에 잘 돌아가던 코드를 뜯어고치지 말아야 한다는 뜻

## LSP: 리스코프 치환 원칙 (Liskov Substitution Principle)

개념: 자식 클래스는 언제나 부모 클래스를 대체할 수 있어야 한다.
이유: 부모 클래스 자리에 자식 클래스를 넣어도 프로그램이 의도대로 돌아가야 상속의 의미가 있습니다.


## ISP: 인터페이스 분리 원칙 (Interface Segregation Principle)
개념: 사용하지 않는 메서드에 의존하도록 강제해서는 안 됨
이유: 거대한 인터페이스 하나보다는, 작고 구체적인 여러 개의 인터페이스가 나음

## DIP: 의존역전 원칙 (Dependency Inversion Principle)
개념: 구체적인 것에 의존하지 말고, 추상적인 것(인터페이스)에 의존하라.
이유: 변화하기 쉬운 구체적인 클래스보다 변화가 없는 인터페이스에 기대야 코드가 흔들리지 않음

|원칙|요약|핵심 키워드|
|---|---|---|
|SRP|한 클래스는 한 가지 일만!|책임 분리|
|OCP|기존 코드 수정 없이 기능 추가!|확장성|
|LSP|자식은 부모 역할을 완벽히 수행!|상속 신뢰성|
|ISP|필요한 기능만 있는 인터페이스!|인터페이스 세분화|
|DIP|구체적인 것 말고 인터페이스를 바라봐!|결합도 낮추기|
