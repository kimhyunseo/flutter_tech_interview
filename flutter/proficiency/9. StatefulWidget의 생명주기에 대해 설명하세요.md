# StatefulWidget의 생명주기에 대해 설명하세요.

## StatefulWidget의 일생 (순서대로)
1. createState()
위젯이 위젯 트리에 삽입될 때 가장 먼저 호출됩니다. 해당 위젯을 위한 **'State 객체'**를 생성하는 단계입니다.

2. initState()
State 객체가 생성된 후 딱 한 번 호출됩니다.

용도: 비동기 데이터를 불러오거나, 애니메이션 컨트롤러 초기화, ScrollController 리스너 등록 등 초기 설정을 수행합니다.

주의: 여기서 BuildContext를 사용하는 기능(예: Navigator, Theme)은 아직 위젯이 완전히 트리에 올라가지 않아 에러가 날 수 있습니다.

3. didChangeDependencies()

initState() 바로 다음에 호출되며, 위젯이 의존하는 InheritedWidget(예: Theme, Locale, Provider)이 변경될 때마다 호출됩니다.

4. build() (가장 자주 호출됨)

화면에 UI를 그리는 단계입니다.

트리거: initState(), didChangeDependencies(), didUpdateWidget(), 그리고 우리가 가장 자주 쓰는 **setState()**가 호출될 때마다 다시 실행됩니다.

주의: 여기서는 오직 UI 렌더링에만 집중해야 하며, 복잡한 로직이나 데이터 요청을 넣으면 앱이 매우 느려집니다.

5. didUpdateWidget()

부모 위젯이 재구성되어 현재 위젯의 **설정(Configuration)**이 바뀌었을 때 호출됩니다. (예: 부모가 넘겨주는 final 변수값이 바뀐 경우)

6. deactivate()

State 객체가 위젯 트리에서 제거될 때 일시적으로 호출됩니다. (거의 쓰이지 않지만, 다른 트리 위치로 이동하기 직전 단계입니다.)

7. dispose() (마지막 단계)

State 객체가 영구적으로 제거될 때 호출됩니다.

용도: AnimationController, TextEditingController, Stream 등을 닫아서 **메모리 누수(Memory Leak)**를 방지하는 청소 단계입니다.

비유: 무대를 다 쓰고 조명을 끄고 나가는 뒷정리팀과 같습니다.

---

| 메서드                          | 호출 시점                            | 특징                                                 |
| ---------------------------- | -------------------------------- | -------------------------------------------------- |
| `createState()`              | StatefulWidget 생성 시              | State 객체 생성                                        |
| `initState()`                | State 객체 생성 직후                   | 초기화, 이벤트 리스너, 컨트롤러 설정 가능. **super.initState() 필수** |
| `didChangeDependencies()`    | 위젯이 의존하는 InheritedWidget이 바뀌었을 때 | 테마, MediaQuery 같은 의존성 초기화                          |
| `build()`                    | UI를 렌더링할 때                       | 화면 업데이트 시 반복 호출 가능                                 |
| `didUpdateWidget(oldWidget)` | 부모 위젯이 rebuild되어 새 위젯이 전달될 때     | 이전 위젯 정보와 비교하여 상태 업데이트 가능                          |
| `setState()`                 | 상태가 바뀔 때 호출                      | build()를 다시 실행시켜 화면 갱신                             |
| `deactivate()`               | 위젯이 트리에서 제거될 때                   | 잠시 제거될 때 호출                                        |
| `dispose()`                  | 위젯이 완전히 제거될 때                    | 컨트롤러 해제, 리소스 정리                                    |

단계|	메서드	|호출 횟수	|주요 |작업|
|-|-|-|-|-|
탄생	|initState	|1회	|컨트롤러 초기화, 데이터 로드 시작|
성장	|didChangeDependencies	|여러 번	|Theme, 언어 설정 등 외부 변화 대응|
활동	|build	|매우 자주	|UI 그리기 (렌더링)|
변화	|didUpdateWidget	|여러 번	|부모로부터 받은 설정값 변경 대응|
죽음	|dispose	|1회	|메모리 해제 (컨트롤러 종료)|

---

메모리 누수 언급: > "저는 특히 dispose() 단계를 중요하게 생각합니다. 사용하지 않는 컨트롤러나 리스너를 적절히 해제하지 않으면 앱의 성능 저하로 이어지기 때문에 반드시 체크합니다."

Context 사용 주의점: > "initState()에서는 context를 통한 Navigator나 Theme 접근이 제한될 수 있습니다. 이를 안전하게 처리하기 위해 didChangeDependencies()나 Future.microtask()를 활용하기도 합니다."

Widget vs State 구분: > "Flutter에서 위젯은 불변(Immutable)이지만, State 객체는 가변(Mutable)입니다. didUpdateWidget이 호출된다는 것은 위젯 틀은 바뀌었지만 상태 객체는 유지되고 있다는 증거입니다."
