# Clean Architecture를 사용하는 이유는 무엇인가요?

클린 아키텍처를 사용하는 가장 큰 이유는 **"소프트웨어가 시간이 지나도 썩지 않게 하기 위해서"**입니다.
단순히 폴더를 예쁘게 나누는 것이 아니라, 변화에 유연하게 대응하고 테스트하기 쉬운 견고한 앱을 만드는 것이 목적이죠. 구체적인 4가지 핵심 이유를 정리해 드립니다.

## 1. 프레임워크로부터의 독립 (Framework Independence)
우리는 Flutter를 사랑하지만, 기술의 세계에 영원한 것은 없습니다.

이유: 클린 아키텍처를 쓰면 비즈니스 로직(심장)이 Flutter(도구)에 의존하지 않습니다. 만약 나중에 핵심 로직을 서버(Dart Backend)로 옮기거나, 심지어 다른 프레임워크로 바꾼다고 해도 도메인 계층의 코드는 단 한 줄도 고칠 필요가 없습니다.

한 줄 요약: "Flutter는 UI를 그리는 도구일 뿐, 우리 서비스의 본질이 아니다."

## 2. 압도적인 테스트 용이성 (Testability)
UI나 데이터베이스가 없어도 비즈니스 로직이 잘 돌아가는지 확인할 수 있습니다.

이유: 도메인 계층은 순수 Dart 코드로만 되어 있습니다. 시뮬레이터를 띄우거나 서버에 연결하지 않고도 **유닛 테스트(Unit Test)**를 통해 "로그인 로직이 정상인가?", "경기 점수 계산이 맞는가?"를 1초 만에 검증할 수 있습니다.

비유: 자동차 엔진이 잘 돌아가는지 확인하려고 매번 차 전체를 조립해서 도로에 나갈 필요가 없는 것과 같습니다.

## 3. 유지보수와 변화에 대한 유연성 (Maintainability)
기획은 언제나 변합니다. "데이터베이스를 Hive에서 SQFlite로 바꿔주세요" 혹은 "API 서버 주소가 바뀌었어요"라는 요청이 와도 두렵지 않습니다.

이유: 데이터 계층(Data Layer)만 수정하면 됩니다. 도메인이나 UI는 건드릴 필요가 없습니다. 영향 범위가 명확히 격리되어 있어 **사이드 이펙트(Side Effect, 예기치 못한 버그)**를 최소화할 수 있습니다.

경기 데이터를 API로 가져오다가 로컬 DB 캐싱을 추가해야 할 때, 데이터 레이어의 레포지토리 로직만 살짝 고치면 UI는 그대로 유지됩니다.

## 4. 대규모 협업과 가독성 (Scalability)
여러 개발자가 동시에 작업할 때 충돌을 방지합니다.

이유: 역할 분담이 명확합니다. "A님은 UI(Presentation)를 만드시고, B님은 API 연동(Data)을 하시고, 저는 비즈니스 로직(Domain)을 짤게요"가 가능해집니다. 각자가 수정하는 파일이 겹치지 않아 효율이 극대화됩니다.
