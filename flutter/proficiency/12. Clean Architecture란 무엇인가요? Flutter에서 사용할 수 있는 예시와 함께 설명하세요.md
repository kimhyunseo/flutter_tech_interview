# Clean Architecture란 무엇인가요? Flutter에서 사용할 수 있는 예시와 함께 설명하세요.
## Clean Architecture란?
**클린 아키텍처(Clean Architecture)**는 소프트웨어를 설계할 때 **"관심사를 분리(Separation of Concerns)"**하여, 비즈니스 로직이 UI, 데이터베이스, 외부 프레임워크 등에 의존하지 않게 만드는 설계 원칙입니다.

로버트 C. 마틴(Uncle Bob)이 제안한 이 방식의 핵심은 **"프레임워크는 도구일 뿐, 서비스의 본질(비즈니스 로직)을 결정해서는 안 된다"**는 것입니다

> 비즈니스 로직을 프레임워크나 외부 의존성으로부터 분리하는 아키텍처 패턴입니다.
핵심 목표는 딱 하나:
의존성 방향을 안쪽으로만 흐르게 만들어서, 핵심 로직을 보호하는 것

## 클린 아키텍처의 3대 계층
Flutter 프로젝트에 적용할 때 보통 다음과 같이 3개 레이어로 나눕니다. 화살표 방향은 의존성의 방향이며, 바깥쪽에서 안쪽(도메인)으로만 향해야 합니다.

### ① Domain Layer (도메인 계층 - 심장부)
가장 순수한 영역입니다. Flutter나 외부 패키지에 의존하지 않는 순수 Dart 코드로만 작성됩니다.

- Entities: 서비스의 핵심 모델 (예: User, Game).
- Use Cases: 비즈니스 로직의 단위 (예: GetGameList, Login).
- Repositories (Interface): 데이터가 어디서 오는지 모르지만, "이런 데이터가 필요해"라고 정의만 해둔 계약서.

### ② Data Layer (데이터 계층 - 일꾼)
도메인에서 정의한 인터페이스를 실제로 구현하는 곳입니다.

- Repositories (Implementation): 도메인 인터페이스를 구현하며, 데이터 소스를 결정합니다.
- Data Sources: 실제로 API를 호출하거나(Remote), 로컬 DB(Local)에 접근합니다.
- Models: JSON 데이터를 Dart 객체로 변환하기 위한 데이터 구조 (Data Transfer Object).

### ③ Presentation Layer (프레젠테이션 계층 - 얼굴)
사용자에게 보여지는 UI와 상태 관리가 일어나는 곳입니다.

- UI (Widgets): Flutter 화면.
- State Management: BLoC, Riverpod, Provider 등을 사용하여 UI 상태를 관리합니다.

## 클린 아키텍처의 득과 실

| 장점 | 단점 |
| --- | --- |
| 테스트 용이성: UI 없이 로직만 테스트 가능 | 복잡도 증가: 단순한 기능도 많은 파일을 만들어야 함 |
| 교체 용이성: DB나 API 구조가 바뀌어도 UI는 안전함 | 학습 곡선: 초기에 구조를 잡는 데 시간이 걸림 |
| 협업 효율: 계층이 나뉘어 있어 역할 분담이 명확함 | 코드 양: 보일러플레이트 코드가 많아짐 |

## 왜 이렇게 복잡하게 나누나요?
1️⃣ 테스트 쉬움
- Domain은 pure Dart라서
- Flutter 없이 unit test 가능
2️⃣ Firebase → Supabase로 바꿔도
- Data Layer만 수정하면 됨
- Domain은 그대로
3️⃣ 유지보수 쉬움
- UI 수정해도 비즈니스 로직 영향 없음

## 정리

"클린 아키텍처는 비즈니스 로직을 UI나 데이터 소스로부터 독립시켜 유지보수와 테스트가 용이한 소프트웨어를 만드는 설계 원칙입니다.

Flutter에서는 주로 Domain, Data, Presentation의 세 계층으로 나누어 구현하며, 특히 **의존성 규칙(Dependency Rule)**을 지켜 안쪽 계층인 도메인이 바깥쪽인 프레임워크나 DB에 의존하지 않게 하는 것이 핵심입니다. 이를 통해 규모가 큰 프로젝트에서도 코드의 정체성을 유지하고 변화에 유연하게 대응할 수 있습니다."
