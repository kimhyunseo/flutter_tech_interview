# ViewModel의 역할은 무엇인가요?
ViewModel은 UI(View)와 비즈니스 로직/상태(Model)를 분리해주는 중간 계층

##  ViewModel의 3가지 핵심 역할
### 1. 상태 관리 (State Management)
UI에서 보여줄 데이터를 저장하고 관리
상태 변경 시 UI에 알림 전달

### 2. 비즈니스 로직 처리 (Business Logic)
API 호출, 데이터 가공, 계산 등
UI에서 직접 처리하지 않고, ViewModel에서 처리 후 결과 전달

"전송 버튼 클릭"이라는 이벤트가 View에서 발생하면, ViewModel은 "입력값 검증 -> API 호출 -> 결과 업데이트"라는 로직을 수행합니다. View는 데이터가 어떻게 서버로 가는지 알 필요가 없습니다.

### 3. UI 로직과의 분리 (Decoupling)
UI는 오직 화면 표시와 사용자 이벤트 처리
ViewModel이 상태/로직을 담당 → 테스트 용이, 재사용 가능

## 추가 질문

### MVVM에서 ViewModel과 Model의 차이
질문 예시:
“ViewModel과 Model의 차이를 설명해 주세요.”

답변:
Model: 데이터 구조와 상태를 나타내는 순수 객체, 로직 없음
ViewModel: Model을 기반으로 UI에 필요한 상태와 로직 제공
UI는 ViewModel과만 상호작용하고 Model은 직접 접근하지 않음

Model = 데이터
ViewModel = 데이터 가공 + 상태 관리
UI(View) = 화면 표시

### 왜 ViewModel을 만들어야 하나요?
질문 예시:
“UI에서 직접 로직을 처리하면 안 되나요?”

답변:
UI와 로직을 섞으면 유지보수 어려움
테스트가 어려워지고, 재사용성 낮음

ViewModel 사용 → 상태 관리 + 테스트 용이 + UI/로직 분리

###  Flutter에서 ViewModel과 Provider/Riverpod의 관계
질문 예시:
“Flutter에서 ViewModel을 상태관리 라이브러리와 함께 쓰는 이유는?”

답변:
Provider/Riverpod → ViewModel 상태를 UI와 연결
상태 변경 시 자동으로 UI에 알림 (ref.watch)
UI는 rebuild만 담당하고, 로직/상태는 ViewModel에서 처리


### "ViewModel과 단순한 'State 관리자'의 차이점이 무엇인가요?"
답변 핵심: "비즈니스 로직의 유무입니다."

상세 답변:
"단순한 상태 관리자는 데이터의 변경을 UI에 알리는 역할에 집중하지만, ViewModel은 해당 데이터를 변경하기 위한 '규칙' 즉, 비즈니스 로직을 포함합니다. 예를 들어 단순히 숫자를 올리는 것은 상태 관리지만, '사용자 권한을 체크하고 API를 호출한 뒤 결과에 따라 다른 상태를 업데이트' 하는 일련의 과정을 책임지는 것이 ViewModel의 역할입니다."


### Provider vs Riverpod, 성능 차이가 있나요?
답변:
작은 앱 → 체감 차이 거의 없음
큰 앱 → Provider는 위젯 전체 리빌드 가능 → 불필요한 빌드 발생
Riverpod → 필요한 위젯만 리빌드 → 성능 최적화 가능
즉, 리빌드 최소화 + 메모리 관리 측면에서 Riverpod이 유리

### Notifier와 AsyncNotifier 차이
"둘의 가장 큰 차이점은 상태 초기화 방식과 관리 형태에 있습니다.

Notifier는 초기화 시 즉시 값을 반환하며, 내부적으로 보관 중인 동기적인 데이터를 다룰 때 주로 사용합니다. 반면 AsyncNotifier는 초기화 과정에서 Future를 반환하여 비동기 작업을 수행할 수 있습니다.

특히 AsyncNotifier는 상태를 AsyncValue로 감싸서 관리하기 때문에, UI 레이어에서 별도의 로딩 처리를 하지 않아도 .when() 메서드를 통해 로딩 중, 에러 발생, 성공 시의 화면을 선언적으로 쉽게 구현할 수 있다는 큰 장점이 있습니다.

Notifier는 동기 상태 관리용이고,
AsyncNotifier는 비동기(API, DB 등) 상태 관리용

#### Notifier란?
📌 특징
동기 데이터 관리 전용
즉시 값이 있는 상태에 사용
로딩/에러 개념 없음
state에 바로 값 저장

📌 사용 예
```
class CounterNotifier extends Notifier<int> {
  @override
  int build() {
    return 0; // 초기값
  }

  void increment() {
    state++;
  }
}

final counterProvider =
    NotifierProvider<CounterNotifier, int>(CounterNotifier.new);
```

📌 언제 사용?
✅ 카운터
✅ 토글 버튼
✅ 탭 선택
✅ 로컬 UI 상태
👉 "바로 쓸 수 있는 값"일 때

#### AsyncNotifier란?
📌 특징
비동기 전용
API / DB / 네트워크 처리
상태가 AsyncValue<T> 형태
로딩/에러/성공 자동 관리
AsyncValue<T>
→ loading / data / error

📌 사용 예
```
class UserNotifier extends AsyncNotifier<User> {
  @override
  Future<User> build() async {
    return await api.fetchUser();
  }
}

final userProvider =
    AsyncNotifierProvider<UserNotifier, User>(UserNotifier.new);
```

📌 상태 구조
```
ref.watch(userProvider)
// → AsyncValue<User>
ref.watch(userProvider).when(
  data: (user) => Text(user.name),
  loading: () => CircularProgressIndicator(),
  error: (e, _) => Text('에러'),
);
```

📌 언제 사용?
✅ API 호출
✅ 서버 데이터
✅ DB 조회
✅ 로그인 정보
👉 "기다려야 하는 값"일 때

✔ Notifier 쓰는 경우
"네트워크 안 쓰고, 앱 내부 상태만 관리할 때"
예:
좋아요 버튼
다크모드 on/off
페이지 인덱스

✔ AsyncNotifier 쓰는 경우
"서버랑 통신하면 무조건"
예:
게시글 목록
유저 정보
검색 결과

"Notifier는 동기적인 상태를 관리할 때 사용하는 Provider로,
로컬 UI 상태나 즉시 사용 가능한 데이터에 적합합니다.
반면 AsyncNotifier는 비동기 작업을 전제로 설계되어 있어서,
API 호출이나 DB 조회처럼 시간이 걸리는 작업을 처리할 때 사용합니다.
AsyncValue를 통해 로딩, 성공, 에러 상태를 자동으로 관리할 수 있다는 점이 가장 큰 차이입니다."
