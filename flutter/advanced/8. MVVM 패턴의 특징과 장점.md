# MVVM 패턴의 특징과 장점은 무엇인가요?
## MVVM의 특징
### 1. 느슨한 결합 (Loose Coupling)

View는 ViewModel을 알지만 ViewModel은 View의 존재를 전혀 모름
UI 디자인이 완전히 바뀌어도 데이터 로직(ViewModel)은 수정할 필요가 없음

👉 Model에 직접 접근 ❌
View → ViewModel → Model
✔ 구조 일관성 유지
✔ 의존성 감소

### 2. 데이터 바인딩 (Data Binding)

View가 ViewModel의 상태를 구독하고 있다가, 상태가 바뀌면 스스로 화면을 갱신
Riverpod의 ref.watch()가 바로 이 데이터 바인딩 역할을 수행하여, 상태 변경 시 UI를 자동으로 리빌드합니다.

#### Flutter & Riverpod의 작동 원리 (관찰과 구독)
Riverpod에서 데이터 바인딩은 '구독(Subscription)' 모델을 따릅니다.
- 관찰(Observe): 위젯이 build() 메서드 안에서 ref.watch(provider)를 호출하는 순간, 해당 위젯은 그 Provider의 구독자(Subscriber) 명단에 이름을 올립니다.
- 알림(Notify): ViewModel(Notifier)에서 state = ...를 통해 상태를 변경하면, Riverpod은 해당 Provider를 구독하고 있는 위젯들에게 **"상태가 변했으니 다시 그려!"**라고 알림을 보냅니다.
- 반응(Reactive): 알림을 받은 위젯은 즉시 자신의 build() 메서드를 다시 실행하여 변경된 상태를 화면에 반영합니다.

### 3. 관심사 분리 (Separation of Concerns)

데이터 로직, UI 로직, 비즈니스 로직이 명확하게 자기 자리에서만 작동

👉 UI와 비즈니스 로직이 섞이지 않음
View → 화면만 담당
ViewModel → 상태/로직 담당
❌ setState 남발 구조 → 유지보수 어려움
✅ MVVM → 구조 깔끔


## MVVM의 장점

### 1. 유닛 테스트(Unit Test)의 용이성

ViewModel은 순수한 Dart 코드로만 이루어져 있어 BuildContext 없이도 테스트가 가능
앱을 직접 실행해서 버튼을 눌러보지 않아도, "로그인 함수를 실행했을 때 상태가 '성공'으로 바뀌는가?"를 코드로 즉시 검증할 수 있음

### 2. 유지보수 및 협업 효율

코드 구조가 규격화되어 있어 여러 개발자가 동시에 작업하기 좋음
한 명은 UI(View)를 다듬고, 한 명은 API 연동 로직(ViewModel/Model)을 짤 수 있어 작업 속도가 빨라짐

### 3. UI와 로직의 독립성

기획 단계에서 UI 디자인이 확정되지 않아도 비즈니스 로직을 먼저 완성할 수 있음
