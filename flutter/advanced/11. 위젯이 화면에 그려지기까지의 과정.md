# 위젯이 화면에 그려지기까지의 과정을 설명해보세요.

## 1️⃣ 위젯 트리 (Widget Tree) – 설계도
정의: 개발자가 작성한 UI 코드 그대로의 트리 구조
역할: 화면 설계도, 위젯의 종류와 구조를 정의

특징:
불변(Immutable)
상태를 직접 가지지 않음
단순히 설정값(config)만 갖고 있음

예시:

MaterialApp
 └ Scaffold
    └ Column
       ├ Text("Hello")
       └ ElevatedButton()

Tip: Widget Tree는 “화면을 그리는 설계도”라고 기억하면 편함

## 2️⃣ 엘리먼트 트리 (Element Tree) – 관리자
정의: 위젯과 화면에 그려지는 객체(RenderObject) 사이를 연결하는 중재자

역할:
BuildContext 제공 → 트리 상의 위치 정보
StatefulWidget의 상태 유지
새 Widget Tree와 기존 Element Tree를 비교(Diffing) → 변경된 부분만 업데이트

특징:
상태(StatefulWidget) 보존
타입(Type)과 Key가 같으면 기존 Element 재사용 → 효율적

예시:
Widget(Text)  →  Element(TextElement)  →  RenderParagraph

Tip: Element Tree는 “화면을 관리하는 관리자”라고 생각

## 3️⃣ 렌더 트리 (Render Tree) – 실무자
정의: 실제 화면에 그려지는 객체 트리

역할: Layout, Paint, Compositing & Rasterizing 단계 수행

렌더링 단계:
① Layout
부모 → 자식: Constraints 전달 (최소/최대 크기)
자식 → 부모: Size 결정
예: Column 위젯 안에 Text와 Button 위치 계산

② Painting
위치, 색상, 테두리 등 픽셀 단위로 그림
Canvas에 그려짐

③ Compositing & Rasterizing
여러 레이어를 합쳐 GPU로 전송
화면에 최종 픽셀 출력


## 상태 변경 시 흐름
상태가 바뀌면 새로운 Widget Tree 생성
Element Tree에서 새 Widget Tree와 비교 → 타입과 Key 확인
같으면 기존 Element 재사용
다르면 새 Element 생성
변경된 부분만 Render Tree 업데이트 → 성능 최적화
전체 화면을 다시 그리지 않고 효율적 갱신 가능


"상태가 변경되면 Flutter는 먼저 새로운 위젯 트리를 생성합니다. 
이때 관리자 역할을 하는 Element 트리가 이전 위젯과 새 위젯의 타입과 키를 비교합니다. 
만약 두 값이 같다면 실제 렌더링을 담당하는 무거운 RenderObject를 새로 만들지 않고 재사용하며, 
바뀐 정보만 업데이트합니다. 이러한 'Diffing 알고리즘' 덕분에 Flutter는 
최소한의 리소스만 사용하여 화면을 효율적으로 갱신하고, 
초당 60프레임 이상의 부드러운 퍼포먼스를 유지할 수 있습니다."
