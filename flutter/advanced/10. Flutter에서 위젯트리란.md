# Flutter에서 위젯트리란 무엇인가요? 
위젯 트리는 Flutter 화면을 구성하는 모든 위젯들이
트리 구조로 연결된 계층 구조

## 1. 위젯 트리란?
Flutter 화면 = 위젯들의 조합
모든 위젯은 부모-자식 관계로 연결됨 → 트리 구조 형성

예:
MaterialApp
 └ Scaffold
    └ AppBar
    └ Column
       ├ Text
       ├ Button
       └ Image

👉 이 구조 전체가 위젯 트리

위젯 트리는 **불변(Immutable)** 입니다. 데이터(상태)가 바뀌면 기존 트리를 수정하는 게 아니라, 새로운 위젯 트리를 통째로 다시 만듬

상태 변경 시:

setState / ref.watch
  ↓
build() 호출
  ↓
새 Widget Tree 생성
  ↓
Diff 비교
  ↓
변경된 부분만 Render

👉 전체 다시 그리는 거 ❌
👉 변경 부분만 갱신 ⭕

## 2. Flutter의 3가지 트리
Widget Tree: "어떤 UI를 만들고 싶은가?"에 대한 설계도 (매우 가볍고 자주 교체됨)

👉 개발자가 작성한 위젯 구조
불변(immutable)
매번 새로 생성됨

Element Tree: "Widget과 RenderObject 사이의 연결 고리". 위젯의 위치 정보(BuildContext)를 가지고 있으며, 실제 화면의 변경된 부분만 찾아내는 중재자 역할을 함

👉 Widget ↔ Render 연결
상태 유지 담당
재사용 관리

Render Tree: "실제로 화면의 어디에, 어떤 크기로 그릴 것인가?"를 담당하는 트리입니다. 가장 무겁고 실제 픽셀을 계산
👉 화면에 픽셀로 그림
layout
paint


Widget Tree → 설계도, 불변, 화면에 직접 없음
Element Tree → 위젯-렌더 객체 연결, 상태 유지, 변경 비교
Render Tree → 실제 화면 그리기
상태 변경 → 새 위젯 생성 → Element가 비교 → 변경된 부분만 Render → 성능 최적화

### 왜 트리를 3개나 사용하나요?
상태가 변할 때마다 무거운 Render Tree를 매번 다시 만드는 것은 성능상 비효율적입니다.

작동 원리: 상태가 변해 Widget Tree가 새로 생성되면, Element Tree가 이전 설계도와 새 설계도를 비교합니다.

결과: "어? 텍스트 내용만 바뀌었네?"라고 판단되면 실제 렌더링 정보(Render Tree)의 해당 부분만 업데이트합니다. 이 메커니즘 덕분에 Flutter는 초당 60~120프레임의 부드러운 속도를 유지할 수 있습니다.

## 추가 질문

### BuildContext가 위젯 트리에서 어떤 역할을 하나요?

답변: "BuildContext는 위젯 트리 내에서 해당 위젯의 위치를 나타내는 핸들이며, 실제로는 Element 트리에 접근할 수 있는 참조값입니다."

### 위젯은 불변(Immutable)이라고 하셨는데, 그럼 어떻게 화면이 바뀌나요?

답변: "위젯 자체는 바뀌지 않지만, 새로운 위젯 트리를 생성하여 이전 트리와 교체하는 방식으로 화면을 갱신합니다. 그 과정에서의 효율성은 Element 트리가 보장해 줍니다."
