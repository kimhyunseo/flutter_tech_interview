# Riverpod 상태관리와 provider 상태관리의 차이는 무엇인가요? 

## Riverpod vs Provider 차이점 요약
1. 위젯 트리 의존성 (가장 큰 차이)

Provider: **위젯 트리에 의존** 데이터를 읽으려면 해당 Provider가 반드시 위젯 트리 상단(부모)에 선언되어 있어야 하며, BuildContext를 통해 위로 거슬러 올라가며 데이터를 찾음

Riverpod: **위젯 트리와 독립적** Provider를 전역 변수처럼 선언하여 어디서든 접근할 수 있음. 데이터를 찾기 위해 트리를 뒤질 필요가 없어 구조가 훨씬 자유로움

2. BuildContext 활용 여부

Provider: **상태에 접근하기 위해 반드시 context가 필요** 이로 인해 UI 로직이 아닌 일반 클래스(Service, Repository)에서 상태를 쓰기가 매우 번거로움

Riverpod: **context 대신 ref 객체를 사용하여 상태에 접근** UI 밖의 비즈니스 로직에서도 ref만 있으면 상태를 자유롭게 읽고 수정할 수 있어 관심사 분리가 훨씬 쉽습니다.


3. 에러 발견 시점 (안전성)

Provider: **런타임 에러에 취약** 상단에 Provider를 주입하지 않고 사용하면 앱이 실행 중 튕김 (빨간 줄이 안 떠서 미리 알기 어려움)

Riverpod: **컴파일 타임 에러** Provider가 선언되지 않았거나 잘못 사용되면 코드를 작성하는 즉시 빨간 줄이 뜨므로, 실행 전에 실수를 바로잡을 수 있음


## 정리
“기존 Provider는 BuildContext에 의존하여 상태를 읽고, UI와 상태 로직이 강하게 결합됩니다.
반면 Riverpod은 BuildContext 없이 ref를 통해 상태를 읽고 쓰며,
필요한 위젯만 리빌드하고, UI와 로직을 분리할 수 있습니다.
덕분에 테스트와 재사용, 유지보수가 쉬워지고, 큰 프로젝트에서도 안정적입니다.”
