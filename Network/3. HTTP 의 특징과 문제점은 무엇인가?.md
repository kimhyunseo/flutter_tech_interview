# HTTP 의 특징과 문제점은 무엇인가?
HTTP는 클라이언트와 서버 간에 데이터를 주고받기 위한 비연결성(Connectionless), 무상태성(Stateless)의 프로토콜

## HTTP의 주요 특징
### 1. 비연결성 (Connectionless)
요청(Request) → 응답(Response) 후
연결을 바로 끊음
👉 계속 연결 유지 안 함

#### 장점
서버 자원 절약
많은 사용자를 동시에 처리 가능

#### 문제점
매 요청마다 연결 설정 비용 발생
성능 저하 가능

→ 그래서 나온 것:
Keep-Alive
HTTP/2
HTTP/3

### 2. 무상태성 (Stateless)
서버가 이전 요청을 기억하지 않음
매 요청은 독립적

#### 장점
서버 확장성 좋음 (Scale-out 용이)
로드밸런싱 유리

#### 문제점
로그인 상태 유지 불가
사용자 정보 기억 못함

→ 그래서 나온 것:
Cookie
Session
JWT 토큰

### 3. 요청-응답 구조 (Request-Response)
클라이언트가 요청해야만 서버가 응답함.
👉 서버가 먼저 보내지 않음

#### 문제점
실시간 통신에 부적합

→ 해결:
WebSocket
SSE

### 4. 텍스트 기반 프로토콜
HTTP 메시지는 텍스트 형식.

#### 장점
사람이 읽기 쉬움
디버깅 용이

#### 단점
바이너리보다 상대적으로 무거움

## HTTP의 구조적 문제점 정리
| 특징          | 문제점           |
| ----------- | ------------- |
| 비연결성        | 요청마다 연결 설정 비용 |
| 무상태성        | 로그인 상태 유지 어려움 |
| 요청-응답 구조    | 실시간 통신 어려움    |
| 평문 전송(HTTP) | 보안 취약         |

## HTTP의 문제점
### 1. 보안 취약
문제: 데이터를 암호화하지 않고 텍스트 그대로 보냅니다. 누군가 중간에서 가로채면 로그인 비밀번호나 개인정보를 다 볼 수 있음 (Sniffing 위험)

해결: HTTPS(HTTP + SSL/TLS)를 사용하여 데이터를 암호화합니다.

### 2. 기억 상실 (Stateless의 단점)
문제: 서버가 나를 기억 못 하니, 페이지를 이동할 때마다 매번 로그인을 다시 해야 하는 번거로움이 생깁니다.

해결: **Cookie(쿠키)** 와 Session(세션), 또는 JWT(JSON Web Token) 같은 토큰 기술을 사용하여 상태를 유지합니다.

### 3. 성능 병목 (HTTP/1.1의 한계)
문제: 한 번에 하나의 파일만 요청할 수 있는 구조(HOL Blocking) 때문에, 웹사이트에 이미지가 많으면 로딩 속도가 현저히 느려집니다. 또한 중복된 헤더 정보를 매번 보내서 데이터 낭비가 심합니다.

해결: HTTP/2 와 **HTTP/3(QUIC)** 가 등장하여 여러 파일을 동시에 받거나(Multiplexing), 헤더를 압축해서 속도를 비약적으로 높였습니다.
