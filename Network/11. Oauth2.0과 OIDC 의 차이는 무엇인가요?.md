# Oauth2.0과 OIDC 의 차이는 무엇인가요?
## 정의
| 용어                        | 정의                                        | 목적                                     |
| ------------------------- | ----------------------------------------- | -------------------------------------- |
| **OAuth 2.0**             | **권한 부여(Authorization) 프레임워크**            | 클라이언트 앱이 사용자 데이터를 접근할 수 있는 권한을 안전하게 부여 |
| **OIDC (OpenID Connect)** | OAuth 2.0 기반의 **인증(Authentication) 프로토콜** | 사용자가 누구인지 확인하고, ID 정보를 안전하게 제공         |

## 제공하는 정보
| 기능     | OAuth 2.0         | OIDC                          |
| ------ | ----------------- | ----------------------------- |
| 사용자 인증 | 직접 제공하지 않음        | 제공 (ID Token)                 |
| 권한 부여  | ✅ Access Token 발급 | ✅ Access Token 발급             |
| ID 정보  | ❌                 | ✅ ID Token (사용자 고유 ID, 이메일 등) |

## 토큰 비교
| 토큰               | 내용             | 목적                        |
| ---------------- | -------------- | ------------------------- |
| **Access Token** | 리소스 서버 접근 권한   | OAuth 2.0에서 데이터 접근에 사용    |
| **ID Token**     | 사용자 정보 포함(JWT) | OIDC에서 사용자 인증 및 정보 확인에 사용 |

## 흐름 비교 (Authorization Code 기준)
- OAuth 2.0
    - 클라이언트 → Authorization Server: 권한 요청
    - Authorization Server → 클라이언트: Authorization Code
    - 클라이언트 → Authorization Server: Access Token 요청
    - 클라이언트 → Resource Server: Access Token으로 데이터 접근

- OIDC
    - OAuth 2.0 과정 그대로 진행 +
    - Authorization Server가 ID Token 발급 → 클라이언트가 사용자 인증 가능
    - Access Token은 권한 부여용, ID Token은 사용자 인증용

## 왜 OIDC가 등장했나요?
OAuth 2.0만으로 로그인을 구현하려고 하니 몇 가지 불편함이 있었습니다.

- 표준화된 사용자 정보 부족: OAuth는 "권한"만 주기 때문에, "이 유저 이름이 뭐야?"라고 물었을 때 구글, 카카오, 네이버가 대답해주는 방식이 다 달랐습니다.

- 비효율성: 유저가 누구인지 알기 위해 Access Token을 받고, 그 토큰으로 다시 "유저 정보 API"를 호출하는 번거로움이 있었습니다.

OIDC는 이를 해결하기 위해 **ID Token (JWT)**이라는 규격을 도입했습니다. 이 토큰만 뜯어보면 서버에 다시 물어보지 않아도 유저의 ID, 이메일, 이름 등을 즉시 알 수 있게 표준화한 것이죠.

## 한줄 요약
- OAuth 2.0: “앱이 내 데이터를 사용할 수 있는 허가증”
- OIDC: “앱이 내 신분을 확인할 수 있는 신분증”
