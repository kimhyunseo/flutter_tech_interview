# JWT란 무엇이며 왜 사용하나요?

## JWT(JSON Web Token)란?
JWT는 **JSON 기반의 토큰(Token)** 으로, 사용자 인증 및 정보 교환을 위해 사용
서버가 발급하고, 클라이언트가 저장했다가 요청 시 포함해서 서버에 전달

## JWT의 구조
JWT는 점(.)으로 구분된 세 부분으로 구성

### 1. 헤더 (Header)
토큰의 타입(JWT)과 사용된 해싱 알고리즘(HMAC SHA256 또는 RSA) 정보

### 2. 페이로드 (Payload)
실제 사용자 정보(Claim)가 담긴 부분
- **Registered Claims**: iss(발급자), exp(만료 시간), sub(주제) 등 표준 클레임
- **Public Claims**: 충돌 방지를 위해 URI 형식으로 정의된 클레임
- **Private Claims**: 서비스에서 임의로 정의한 클레임(예: user_id, role)

### 3. 시그니처 (Signature)
헤더와 페이로드를 비밀 키로 해싱하여 위변조 방지

## JWT가 사용되는 이유
### 1. stateless(무상태성)
서버가 사용자 정보를 매번 DB에서 조회할 필요 없이, 토큰만 보고도 인증 가능
→ 서버 부하 감소 및 확장성 향상

### 2. 보안성
시그니처로 토큰의 위변조를 방지
페이로드에 민감 정보를 담지 않고, 필요시 암호화 가능

### 3. 확장성
다양한 서비스(서버, 앱)에서 동일한 토큰 사용 가능
클레임에 권한 정보를 담아 API 접근 제어 용이

## JWT의 단점
### 1. 토큰 탈취 시 보안 문제
토큰이 탈취되면 공격자가 사용자 행세 가능
→ 짧은 만료 시간 설정 및 refresh token 사용으로 보완

### 2. 토큰 크기
페이로드에 많은 정보를 담으면 토큰 크기 증가
→ 매 요청마다 전송 비용 발생

### 3. 수정 불가
토큰이 발급되면 내용을 수정할 수 없음
→ 권한 변경 시 토큰 재생성 필요

## 정리
JWT는 사용자 인증과 권한 부여를 위한 **경량의 안전한 토큰**
**stateless**한 특성으로 서버 부하 감소 및 확장성 향상
**시그니처**로 위변조 방지, **클레임**으로 권한 관리
**짧은 만료 시간**과 **refresh token**으로 보안 강화
REST API 환경에서 **인증과 권한 부여**를 위한 표준으로 널리 사용됨
