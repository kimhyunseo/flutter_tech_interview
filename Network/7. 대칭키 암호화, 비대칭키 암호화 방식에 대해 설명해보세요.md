# 대칭키 암호화, 비대칭키 암호화 방식에 대해 설명해보세요.

## 대칭키(Symmetric Key) 암호화
암호화할 때와 복호화(풀 때)할 때 같은 열쇠를 사용

개념: 송신자와 수신자가 동일한 하나의 키를 공유

장점: 암호화 및 복호화 속도가 매우 빠름. 대용량 데이터를 주고받을 때 효율적

단점(치명적): **'키 전달의 문제'** 가 있음. 상대방에게 열쇠를 보내야 하는데, 그 과정에서 해커가 열쇠를 가로채면 암호화가 아무 소용이 없게 됨

비유: 하나뿐인 열쇠를 가진 보석 상자. 상자를 잠근 사람과 여는 사람이 같은 열쇠를 써야 함

대표 알고리즘: AES, DES

## 비대칭키(Asymmetric Key / 공개키) 암호화
암호화하는 열쇠와 복호화하는 열쇠가 서로 다름

개념: **공개키(Public Key)**와 개인키(Private Key) 한 쌍을 사용

공개키: 누구나 가질 수 있는 열쇠. 데이터를 암호화할 때 씀

개인키: 나만 가진 비밀 열쇠. 암호화된 데이터를 풀 때만 씀

장점: 열쇠를 도둑맞을 걱정이 없음. 공개키는 누구에게나 뿌려도 되고, 데이터를 풀 수 있는 개인키는 나만 가지고 있으면 되니까 보안성이 매우 높음

단점: 복잡한 수학적 계산이 필요해서 대칭키보다 훨씬 느림

비유: 우체통. 누구나 편지를 넣을 수 있지만(공개키), 편지함을 열어 내용을 볼 수 있는 건 집주인(개인키)뿐

대표 알고리즘: RSA, ECC

## 추가 질문
### HTTPS에서는 왜 두 방식을 혼합해서 사용하나요?

질문 의도: 각 암호화 방식의 장단점을 파악하고, 실무적인 해결책(하이브리드 암호화)을 알고 있는지 확인합니다.

핵심 답변: 비대칭키는 키 공유가 안전하지만 연산이 느리고, 대칭키는 속도가 빠르지만 키 전달이 위험합니다. 따라서 비대칭키로 대칭키를 안전하게 공유한 뒤, 실제 데이터는 대칭키로 암호화하여 보안과 속도를 모두 잡기 위함입니다.

### 비대칭키 암호화는 암호화 외에 '전자서명'에도 쓰입니다. 전자서명의 원리는 무엇인가요?

질문 의도: 비대칭키의 방향성(개인키로 암호화 → 공개키로 복호화)을 이해하고 있는지 확인합니다.

핵심 답변: 보낼 데이터의 해시값을 나의 개인키로 암호화하는 것이 서명입니다. 수신자가 나의 공개키로 복호화에 성공한다면, "이 데이터는 확실히 내가 보낸 것임(인증)"과 "내용이 변하지 않았음(무결성)"을 동시에 증명할 수 있습니다.

| 항목     | 비대칭키 암호화                    | 전자서명                      |
| ------ | --------------------------- | ------------------------- |
| 목적     | 데이터 **기밀성** 확보              | 데이터 **무결성 + 송신자 인증**      |
| 사용 키   | **공개키**로 암호화 → **개인키**로 복호화 | **개인키**로 서명 → **공개키**로 검증 |
| 결과     | 암호화된 데이터                    | 서명 (암호화된 해시)              |
| 데이터 접근 | 오직 개인키 소유자만 복호화 가능          | 누구나 공개키로 검증 가능            |
