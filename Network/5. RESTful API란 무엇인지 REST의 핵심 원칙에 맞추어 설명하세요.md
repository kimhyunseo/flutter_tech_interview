# RESTful API란 무엇인지 REST의 핵심 원칙에 맞추어 설명하세요.
RESTful API란 로이 필딩(Roy Fielding)이 제안한
**REST(Representational State Transfer)** 라는 아키텍처 스타일의 제약 조건을 엄격하게 따르는 API

쉽게 말해, **"자원(Resource)을 이름으로 구분하여, 그 상태(State)를 주고받는(Transfer) 방식"**입니다.

## REST의 6가지 핵심 원칙

### 1. 균일한 인터페이스 (Uniform Interface)

가장 중요한 원칙으로, 자원(Resource)을 조작하는 방식이 일관되어야 함

자원 식별: 모든 자원은 고유한 URI를 가짐 (예: /stadiums/1)
표현을 통한 자원 조작: 클라이언트가 JSON, XML 등 적절한 포맷으로 자원을 전송하면 서버는 이를 처리
자기 서술적 메시지: 메시지만 보고도 이 요청이 무엇을 의미하는지 알 수 있어야 함 (HTTP Method와 Header 활용)
HATEOAS: 응답 속에 다음 단계로 갈 수 있는 링크를 포함해야 함 (사실 이 부분이 가장 지키기 어려워 'Restish'에 머무는 경우가 많음)

### 2. 무상태성 (Stateless)

개념: 서버는 클라이언트의 상태(세션 등)를 기억하지 않음
이유: 각 요청은 독립적이어야 하며, 요청에 필요한 모든 정보(토큰 등)를 포함해서 보내야 함
덕분에 서버는 들어오는 요청을 아무 서버나 처리할 수 있어 확장성이 좋아짐

### 3. 클라이언트-서버 구조 (Client-Server Separation)

개념: 사용자 인터페이스(Client)와 데이터 관리(Server) 역할 명확히 나눔
이유: 서로의 내부 구현을 몰라도 되므로, 클라이언트는 UI에 집중하고 서버는 비즈니스 로직에만 집중하여 독립적인 진화 가능

### 4. 캐시 가능 (Cacheable)

개념: HTTP의 기존 인프라를 활용하여 응답 데이터를 캐싱할 수 있습니다.
이유: 같은 요청에 대해 서버에 계속 물어보지 않고 저장된 데이터를 써서 응답 시간을 단축하고 서버 부하를 줄입니다.

### 5. 계층화 시스템 (Layered System)

개념: 클라이언트는 서버와 직접 통신하는지, 중간에 프록시나 로드 밸런서가 있는지 알 수 없습니다.
이유: 보안을 강화하거나 로드 밸런싱을 통해 시스템 안정성을 높이기 위함입니다.

### 6. Code on Demand (선택 사항)

개념: 서버가 클라이언트에 실행 가능한 코드를 보내서 기능을 확장할 수 있습니다. (예: 자바스크립트)

## RESTful API란?
위의 REST 원칙을 지키며 설계된 API를 RESTful API라고 합니다.

URI는 자원 중심 설계
HTTP 메서드로 행위 표현
무상태성 유지
일관된 인터페이스 제공
